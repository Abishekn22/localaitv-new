## Answers to Your 9 Open Questions

### Q1: Can the same `content_id` be edited and approved again after a webhook was already sent?

**Answer for v1: NO.** Once approved and webhook sent, the content_id is locked.

**Recovery path for post-approval corrections:**

```
Super admin clicks "Cancel processing"
  ↓
DELETE /webhook/cancel-job/{job_id}
  ↓
Your pipeline:
  - If task not picked up by Celery → remove from RabbitMQ → 200 OK
  - If task in progress → Celery SIGTERM → 200 OK
  - If task complete → 410 Gone
  ↓
Content reverts to status='pending' in Admin
Admin edits → re-approves → new webhook with approved_version=2
```

**For Phase 2:** Native re-approval flow with explicit supersession. Deferred until we see real usage need.

### Q5: If the Admin Dashboard retries a webhook, will it resend the exact same body and signature?

**Answer: YES — byte-for-byte identical body + same HMAC signature on every retry.**

**Implementation guarantees:**

```python
# Admin Dashboard webhook sender (pseudocode)
def send_webhook(payload):
    # JSON serialized with sort_keys=True for deterministic output
    body_bytes = json.dumps(payload, sort_keys=True).encode('utf-8')
    
    # HMAC signs the raw body bytes
    signature = hmac.new(SHARED_SECRET, body_bytes, hashlib.sha256).hexdigest()
    
    headers = {
        'Content-Type': 'application/json',
        'X-LocalAITV-Signature': signature,                  # SAME on retry
        'X-LocalAITV-Idempotency-Key': payload['idempotency_key'],  # SAME on retry
        'X-LocalAITV-Attempt': str(attempt_number),          # INCREMENTS (not signed)
    }
    
    response = requests.post(WEBHOOK_URL, data=body_bytes, headers=headers)
```

**Key design choices:**
- The signed body does NOT contain timestamps (would break idempotency)
- The attempt counter is in a separate header (not in signed body)
- The idempotency_key is part of the signed body (ensures retry consistency)

Your verification can rely on: same body bytes → same HMAC. Any retry that arrives with a different signature for the same idempotency_key indicates tampering — reject with 401.

### Q7: Will webhook retries resend the exact same raw body?

**Answer: YES — see Q5 for the detailed contract.**

Your verification logic should:
1. Read the raw body bytes from the HTTP request (NOT re-serialized JSON)
2. Compute HMAC-SHA256 over those exact bytes
3. Compare to the `X-LocalAITV-Signature` header
4. Reject if mismatch

Important note: Some HTTP frameworks re-serialize JSON before exposing it to handlers. **Use the raw body**, not framework-deserialized JSON. In FastAPI this means using `await request.body()` directly.

### Q8: Who owns thumbnail generation: Admin Dashboard or AI Pipeline?

**Answer: SPLIT OWNERSHIP based on the use case.**

| Thumbnail | Generated by | When | Purpose |
|---|---|---|---|
| Photo thumbnail (for citizen-uploaded photos) | Admin Dashboard | On upload, via AWS Lambda + Cloudflare Images | Admin queue preview |
| Video preview frame (for citizen-uploaded videos) | Admin Dashboard | On upload, via FFmpeg single-frame extract | Admin queue preview |
| Final bulletin thumbnail (`thumbnail.jpg` in your folder structure) | **AI Pipeline (you)** | After bulletin assembly | Live broadcast preview |
| Individual processed item thumbnail | **AI Pipeline (you)** | After processing | App display |

**Rationale:**
- Admin needs to SEE the citizen upload IMMEDIATELY for review — so Admin generates lightweight previews
- AI Pipeline produces the FINAL broadcast video — so AI generates the broadcast thumbnail (matches video output quality + branding)

**For your scope:** Your existing `video_builder.py` should output `thumbnail.jpg` as part of the standard output bundle (per your proposed folder structure).

### S3 lifecycle configuration (translated from your R2 spec)

In v1.2, storage moved to **AWS S3 (Mumbai)**. The lifecycle policy translates directly — AWS provides richer storage classes (Standard / IA / Glacier IR / Deep Archive) for tiered cost reduction:

```yaml
# AWS S3 lifecycle configuration (per your retention recommendations)
rules:
  - id: bulletin-segments
    prefix: ai-processed-bulletins/*/segments/
    expiration:
      days: 1                        # 24 hours (DELETE)

  - id: bulletin-final
    prefix: ai-processed-bulletins/*/final_bulletin.mp4
    expiration:
      days: 2                        # 48 hours (DELETE)

  - id: bulletin-thumbnail
    prefix: ai-processed-bulletins/*/thumbnail.jpg
    expiration:
      days: 2                        # 48 hours (DELETE)

  - id: bulletin-manifest
    prefix: ai-processed-bulletins/*/bulletin_manifest.json
    transitions:
      - days: 7
        storage_class: STANDARD_IA   # Move to cheaper tier
      - days: 30
        storage_class: GLACIER_IR    # Cheaper archival
    expiration:
      days: 365                      # Keep for 1 year audit

  - id: bulletin-metadata
    prefix: ai-processed-bulletins/*/metadata.json
    transitions:
      - days: 30
        storage_class: GLACIER_IR
    expiration:
      days: 365

  - id: temp-cleanup
    prefix: temp/
    expiration:
      days: 1                        # 24 hours

  - id: ai-processed-tiering
    prefix: ai-processed/
    transitions:
      - days: 8
        storage_class: STANDARD_IA   # 50% cheaper
      - days: 31
        storage_class: GLACIER_IR    # 75% cheaper
      - days: 91
        storage_class: DEEP_ARCHIVE  # 95% cheaper (for compliance retention)
```

This achieves the same retention behavior as your R2 spec, but with **additional cost savings** because S3 has more storage tiers. The intent (delete bulletins after 48 hrs, keep manifests for audit) is preserved exactly.
